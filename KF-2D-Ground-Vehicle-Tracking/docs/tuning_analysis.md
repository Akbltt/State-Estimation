# Tuning Analysis

## Purpose

Assess how process noise (`Q`) and measurement noise (`R`) tuning changes filter behavior in terms of responsiveness, smoothness, and stability.

## Scenarios Included

- Baseline tuning
- Low process noise (`Q` too small)
- High process noise (`Q` large)
- Low measurement noise (`R` small)
- High measurement noise (`R` large)
- No model mismatch reference case

Results are generated by `src/run_tuning_study.m` and saved under `results/tuning_study/`.

## Practical Interpretation

### Process Noise (`Q`)

Low `Q`:

- estimator is smooth but less responsive to real maneuvers
- lag and larger transient errors during acceleration/turning
- can become overconfident in model assumptions

High `Q`:

- estimator reacts quickly to trajectory changes
- follows dynamics better during mismatch periods
- noisier estimates and potentially less smooth velocity states

### Measurement Noise (`R`)

Low `R`:

- estimator trusts measurements strongly
- fast correction, but more measurement noise leakage
- trajectory can look jittery

High `R`:

- estimator trusts model more than measurements
- smoother output and slower correction
- larger bias/lag when model mismatch is present

## Stability and Consistency Cues

Use these checks while tuning:

- compare error envelopes vs `+-3 sigma` bounds
- monitor covariance convergence trends
- check if error spikes align with maneuver sections
- validate that tuning does not produce persistent divergence

## Recommended Workflow

1. Start with realistic sensor variance in `R`
2. Increase `Q` until maneuver lag is acceptable
3. Reduce `Q` if estimates become too noisy
4. Re-run mismatch scenario and compare RMSE and time histories

This keeps tuning grounded in expected operating conditions instead of only minimizing one scalar metric.
